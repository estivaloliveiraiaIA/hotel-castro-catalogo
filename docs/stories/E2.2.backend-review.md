## Story
**As a** time de desenvolvimento do Castro's Park Hotel,
**I want** uma revisão técnica completa do backend (APIs Vercel + schema Supabase),
**so that** a base do sistema seja segura, eficiente e preparada para receber as próximas features (F2 Concierge IA, F7 Eventos, F8 Parceiros).

## Acceptance Criteria
1. **Security audit** das APIs admin: autenticação, proteção contra injection, headers de segurança
2. **Schema Supabase** revisado: tipos de dados corretos, índices necessários, constraints, RLS policies
3. **API endpoints** revisados: tratamento de erros consistente, validação de inputs, responses padronizados
4. **Upload de imagens** revisado: validação de tipo/tamanho, naming seguro, bucket permissions
5. **Variáveis de ambiente** auditadas: nenhuma credencial exposta no bundle frontend
6. **Relatório de melhorias** entregue: lista priorizada de o que corrigir agora vs. o que pode aguardar
7. **Documento de arquitetura atualizado** refletindo a stack real (Vercel + Supabase)

## Dev Notes
### Stack real (vs. documentado)
```
Documentado:          Real:
GitHub Pages    →     Vercel (deploy + serverless functions)
JSON estático   →     Supabase PostgreSQL
Sem backend     →     api/admin/* (7 endpoints Vercel Functions)
GitHub Actions  →     vercel --prod via CLI
```

### Estrutura de arquivos backend
```
api/
  _lib/
    auth.js           # Token: base64("admin:{ADMIN_PASSWORD}")
    supabase.js       # createClient(VITE_SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY)
  admin/
    login.js          # POST — verifica senha, retorna token
    stats.js          # GET — conta registros (4 tabelas)
    places.js         # GET/POST/PUT/DELETE — tabela places
    events.js         # GET/POST/PUT/DELETE — tabela events
    itineraries.js    # GET/POST/PUT/DELETE — tabela itineraries + itinerary_places
    partners.js       # GET/POST/PUT/DELETE — tabela partners
    upload.js         # POST — base64 → Supabase Storage bucket hotel-images
```

### Variáveis de ambiente
```
VITE_SUPABASE_URL          # Pública (injetada no bundle via Vite)
VITE_SUPABASE_ANON_KEY     # Pública (injetada no bundle via Vite)
SUPABASE_SERVICE_ROLE_KEY  # Secreta (apenas server-side)
ADMIN_PASSWORD             # Secreta (apenas server-side)
```

### Pontos de atenção conhecidos
- Autenticação admin é simples (base64) — suficiente para MVP mas não para produção multi-usuário
- Sem rate limiting nos endpoints admin
- Sem validação de tamanho de arquivo no upload
- RLS policies do Supabase precisam ser verificadas

### Testing

- Verificar headers de resposta (Content-Type, CORS)
- Testar upload com arquivo inválido (não-imagem)
- Verificar logs de erro no Vercel
